# Council Session: 2026-06-17 — Yaride

Проект: **Yaride** — Telegram-бот карпулинга для Ярославля и окрестностей (https://github.com/MightyXander/Yaride). Двусторонний маркетплейс, MVP ~5-6 недель, соло-разработка.

## Council Members
- **Head of Growth / Marketplace** — cold-start, ликвидность, геоплотность, каналы, сетевые эффекты.
- **Product / UX Lead** — онбординг, бот vs Mini App, модель ролей, доверие, удержание.
- **Монетизация / CFO** — бизнес-модель без платежей, unit-экономика, burn, дисинтермедиация.

## Context (snapshot)
Стек: Python/aiogram, миграция SQLite→PostgreSQL (v12), FastAPI-админка, Redis (opt-in), Telegram Mini App (React/TS via Lovable), уход с платных Яндекс API на бесплатные Leaflet/OSRM, Railway+Docker. Реализованы: роли, поездки/брони/отмены, календарь, промежуточные остановки, рейтинги/баны, Mini App. Ограничения: без платежей, без GPS, региональный охват. Контекст рынка: уход BlaBlaCar из РФ (2022). **Аналитики продукта в коде нет.** Открыт issue #16 (показать контакт водителя после брони).

## Концепция основателя (приоритет, перебивает советы экспертов)
**Yaride — поиск регулярных поездок ВНУТРИ Ярославля (intracity-коммьютинг), чтобы разгрузить общественный транспорт.** Это не межгород. Решение принято основателем 2026-06-17.

Следствия для рекомендаций ниже:
- **Промежуточные остановки — ядро продукта, НЕ замораживать.** Регулярный городской маршрут, подбирающий людей по пути — это суть концепции (аналог маршрутки). Раз работают — оставляем и развиваем.
- **Шаблоны/регулярные рейсы и алерты на маршрут поднимаются в топ** — это прямой инструмент «разгрузить транспорт».
- **Дисинтермедиация (issue #16) при регулярных поездках опаснее** — одни и те же люди на одном маршруте быстро уйдут в личку. Контакт — через внутрибот-чат / отложенно.

См. раздел «Parked / не приоритет» в конце — туда вынесена гипотеза межгорода.

## Head of Growth / Marketplace
- **Переопределение рынка [ОТКЛОНЕНО основателем, см. Parked]:** эксперт считал, что внутригородской карпулинг обречён, а дыра после BlaBlaCar — межгород. Основатель сознательно выбрал intracity-концепцию (разгрузка транспорта). Гипотеза межгорода вынесена в «Parked / не приоритет».
- **Cold-start:** качать сначала **предложение (поездки водителей)** — размещение поездки имеет нулевую маржинальную стоимость; пассажир уходит навсегда за 30 сек, увидев пустоту.
- **Геоплотность:** один «золотой» маршрут (кандидат — Ярославль↔Рыбинск), а не весь регион. Порог: <3 поездок/день = мёртв, 5-10/день = «дышит».
- **Каналы:** существующие VK/TG попутка-чаты, VK-паблики, Avito, публичный канал маршрута с автопостингом. Отбросить платную рекламу.
- **Аналитика — приоритет #1:** инструментировать события (search, search_result_shown, booking и т.д.). North Star — завершённые совместные поездки/неделя.
- **Виральность:** deep-link шеринг поездки (top impact/effort), публичный канал маршрута, inline-режим, рефералка.
- **Резать сейчас:** платежи, доводку Mini App, промежуточные остановки/OSRM, рейтинги, Redis — преждевременно.

## Product / UX Lead
- **Главная ошибка — два UX-стека параллельно** (бот aiogram + Mini App React) на соло-ресурсе. Каждая фича пилится дважды → рассинхрон. Выбрать один primary; решение требует данных (которых нет).
- **Онбординг:** убрать преждевременный выбор роли на старте; just-in-time регистрация (искать без регистрации, профиль — только на брони).
- **Модель ролей переусложнена:** роль — свойство *поездки*, а не *человека*. Жёсткий тумблер + «смена только без активных поездок» создаёт тупики, воспринимаемые как баг.
- **Доверие — дешёвый рычаг:** профиль/рейтинг водителя на карточке *до* брони; issue #16 — в топ; проверить, что порог рейтинга не блокирует новичков.
- **Удержание:** алерты на маршрут при пустом поиске («сообщим, когда появится») — превращают отток в захват спроса; шаблоны/повтор поездок для водителей.
- **Аналитика — предусловие всего** (повтор тезиса Growth).
- **Переусложнено для стадии:** промежуточные остановки, обязательная причина отмены (сделать пресеты).

## Монетизация / CFO
- **Это портфолио (~70%), не бизнес.** Намеренно вырезан денежный поток — единственный встроенный механизм монетизации маркетплейса (take rate + lock-in, как эскроу BlaBlaCar).
- **«Без платежей» — выбор, а не данность:** Telegram Payments/ЮKassa/СБП/Stars делаются за день. Без них верхняя строка P&L = 0 при любом объёме.
- **Монетизация преждевременна** на порядок. Порог для разговора о деньгах: ~1000+ завершённых поездок/мес + положительный retention.
- **Unit-экономика:** транзакция = завершённая поездка. На межгороде take rate 10-15% ≈ 50-130 ₽/поездка; местные — копейки. Водители подписку платить не будут.
- **Burn несущественен:** ~1-2.5 тыс ₽/мес сейчас, 3-5.5 тыс при 1000 активных. Уход с Яндекса экономит ~0 ₽ сейчас, но снимает будущий variable cost и юридическую привязку. **OSRM не бесплатен — оплачивается в RAM; проверить, что в проде self-hosted, а не демо-ToS-сервер.**
- **Issue #16 = усиление дисинтермедиации (экзистенциальный риск):** раскрытие контакта → пара уходит в личку навсегда. Рекомендация: отложенное раскрытие или внутрибот-чат.
- **Прочие риски:** фрод/no-show без скин-ин-зе-гейм, регуляторика при появлении платежей (152-ФЗ, налоги), bus factor=1, нет лицензии в репозитории.

## Consensus (все трое согласны)
1. **Аналитики нет — это дыра №1.** Все три эксперта независимо назвали инструментирование событий приоритетом, который разблокирует все остальные решения. Стоит ~0 ₽.
2. **Ликвидность важнее фич.** Продукт — liquidity-gated; пустой маркетплейс не лечится никаким UX/маркетингом. Фокус на 1-2 коридорах.
3. **Монетизация преждевременна.** Сейчас цель — ликвидность и retention, не выручка.
4. **Часть фич переусложнена для стадии:** доводка Mini App (паритет с ботом), рейтинги/пороги, Redis — заморозить. _(Промежуточные остановки исключены из заморозки решением основателя — они ядро intracity-концепции, см. выше.)_
5. **Issue #16 — критический узел.** Но единодушно: раскрывать контакт нужно осторожно (поздно / внутренний чат), а не «в лоб».

## Disagreements
| Вопрос | Growth | Product/UX | CFO |
|--------|--------|-----------|-----|
| **Issue #16** | Закрывать обязательно — узел доверия и завершения сделки | В топ-1 — закрывает петлю «забронировал→связался→еду» | Опасен — удаление последнего барьера утечки; раскрывать максимально поздно / внутрибот-чат |
| **Mini App** | Заморозить — переинвестировано, bot-flow + share-ссылки достаточно | Возможно сделать *основным* (карта/маршруты ложатся на rich UI), но решать по данным | Нейтрально — burn несущественен, вопрос не финансовый |
| **Платежи** | Не делать — отсутствие = плюс (меньше трения, нет регуляторики) | Не затрагивает | Главный структурный пробел — без них это не бизнес; вернуть, если метить в стартап |
| **Промежуточные остановки** | Отвлечение, заморозить | Самая переусложнённая фича, заморозить доводку | (не акцентировал) |

**Ключевое противоречие — issue #16:** Growth и Product видят его как драйвер доверия/конверсии, CFO — как механизм дисинтермедиации, убивающий экономику. Примирение: закрыть петлю доверия (контакт нужен), но **через отложенное раскрытие или внутренний чат**, а не прямой Telegram-контакт сразу.

## Parked / не приоритет
- **Гипотеза межгорода (Growth):** запуск на коридоре Ярославль↔Рыбинск/Кострома/Москва, bootstrap предложения, «золотой маршрут». Отклонено в пользу intracity-концепции, оставлено как возможное направление на будущее.
- **Возврат платежей (CFO):** релевантно, только если проект пойдёт в сторону полноценного бизнеса/стартапа. Для текущей концепции — не приоритет.

## Decisions
- 2026-06-17: концепция = **intracity-коммьютинг** (разгрузка транспорта), межгород отклонён.
- 2026-06-17: промежуточные остановки сохраняются и развиваются.
- 2026-06-17: **судьба интерфейса решена (по убеждению, не по данным): Mini App = основной и единственный полноценный интерфейс.** Бот сводится к: точка входа (deep-link), уведомления (бронь подтверждена/отменена, напоминания), read-only просмотр сведений о поездке. Интерактивные сценарии (поиск, создание поездки, бронь, выбор/смена роли) переносятся из бота в Mini App, лишние хендлеры aiogram вычищаются. _Следствие: это рефактор; аналитика (п.1) теперь нужна для оптимизации воронки, а не для выбора интерфейса._
- 2026-06-17: к внедрению — пункты 1-6 выше (начать с аналитики).
