# Этап 0 — Гигиена и основания

**Зависимости от других этапов:** нет (это первый этап).
**Roadmap:** см. `2026-05-11-audit-and-roadmap.md`, раздел 3.
**Покрывает находки:** H1, M9 (часть), L13 (часть), L14, L15.

---

## 1. Цель

Зафиксировать дисциплину, на которой держатся следующие этапы:

- ни одного секрета в репозитории и понятная политика их хранения;
- автоматический lint/format, чтобы стиль не «расходился» в дальнейших рефакторингах;
- единая точка для UX-констант и среды;
- фундамент для версионных миграций (`schema_version`) — этап 1 будет на него опираться.

## 2. Скоуп

### В скоупе

- Ротация `BOT_TOKEN` через BotFather (вручную владельцем).
- Удаление `.env` из git-истории через `git filter-repo` (вручную владельцем; план фиксирует команду и проверку).
- `.gitignore`: добавить `.env`, оставить `yaride.db`.
- Создать `.env.example` с пустыми значениями и комментариями.
- Обновить `AGENTS.md`: новая политика `.env`, инструкции по ротации, ссылка на `.env.example`.
- Подключить `ruff` (lint + format) с конфигом, расширяющим стандартные правила минимально (line-length, double-quotes, isort). Запустить `ruff check --fix` и `ruff format` одним коммитом по всему репо.
- Завести таблицу `schema_version` в SQLite и записать туда текущую версию схемы. Поведение `init_schema`: если запись есть — миграции не дёргать; если нет — выставить текущую версию (для существующих баз) или прогнать миграции с нуля (для новых).
- Перенести UX-константы в `app/config.py`:
  - `SEATS_CHOICES = (2, 3, 4)`
  - `PRICE_CHOICES = (100, 150, 200)`
  - `TIME_STEP_MINUTES = 30`
  - `WORK_HOURS = (6, 23)`
  - `GEO_SUGGEST_LIMIT = 5`
  - `GEO_SUGGEST_MAX_KM = 85.0`
  - `LOCALITY_GEO_MAX_KM = 150.0`
  - `RATING_PROMPT_INITIAL_DELAY_S = 45`
  - `RATING_PROMPT_INTERVAL_S = 180`
- Базовое улучшение логов: формат `"%(asctime)s %(levelname)s %(name)s [%(filename)s:%(lineno)d] %(message)s"`. Без структурного JSON и без correlation id — это уйдёт в Этап 1 для критичных путей.

### Не в скоупе

- Полный structured logging — Этап 1.
- Атомарные миграции с откатом — Этап 1.
- Любое изменение поведения handler'ов.
- Любые новые зависимости кроме `ruff`.

## 3. Поведение после этапа

- `git ls-files .env` возвращает пустую строку (файл не отслеживается).
- `.env.example` присутствует в репо и виден в `README.md` как шаг настройки.
- `ruff check` проходит без ошибок.
- `ruff format --check` проходит без изменений.
- `py -3 -m unittest discover -s tests -v` проходит без изменений (32+ тестов зелёные).
- При первом запуске на чистой машине бот создаёт `yaride.db` с `schema_version=N`, где `N` — текущая версия схемы.
- При запуске на существующем `yaride.db` без `schema_version` бот ставит `schema_version=N` и не дёргает миграции повторно.
- `bot.py` импортирует константы из `app/config.py`, а не определяет их у себя.

## 4. Архитектура изменений

### 4.1 `app/config.py`

`Settings` расширяется именованными константами (через `dataclass` поля с дефолтами, переопределяемыми из env по принципу «прочитал → проверил → подставил»). Для коллекций (`SEATS_CHOICES`, `PRICE_CHOICES`) допускается константа модуля; для скаляров — поле `Settings`.

Загрузка из `.env`:

```text
SEATS_CHOICES=2,3,4
PRICE_CHOICES=100,150,200
TIME_STEP_MINUTES=30
WORK_HOURS_START=6
WORK_HOURS_END=23
GEO_SUGGEST_LIMIT=5
GEO_SUGGEST_MAX_KM=85.0
LOCALITY_GEO_MAX_KM=150.0
RATING_PROMPT_INITIAL_DELAY_S=45
RATING_PROMPT_INTERVAL_S=180
```

Конкретный механизм парсинга списков — деталь плана; принцип: некорректное значение в `.env` → `RuntimeError` на старте (как уже сделано для `BOT_TOKEN`).

### 4.2 `app/db.py` — таблица `schema_version`

```sql
CREATE TABLE IF NOT EXISTS schema_version (
    id INTEGER PRIMARY KEY CHECK(id = 1),
    version INTEGER NOT NULL
);
```

Логика на старте:

1. Если строки нет — это первый запуск или старая база.
2. Если в БД уже есть таблицы `users`, `trips`, `route_points` и нет `schema_version` — считаем, что это старая база на «последней миграции»; ставим текущую версию без выполнения миграций. Это безопасно, потому что миграции уже идемпотентны, и текущий код на текущей схеме работает.
3. Если таблиц нет — `init_schema` создаёт всё с нуля и ставит текущую версию.
4. В дальнейшем (Этап 1) на основе версии будет выполняться линейный набор миграций.

Текущая версия в этом этапе — фиксированное число (`SCHEMA_VERSION = 1`). Этап 1 заменит подход на линейный список.

### 4.3 Линтер

`pyproject.toml` (новый файл) с минимальной конфигурацией `[tool.ruff]`:

- target-version py311 (или то, что реально требуется проектом — определит план),
- line-length 120,
- двойные кавычки,
- isort включён.

Правила, которые гарантированно нарушаются текущим кодом и которые не выгодно править сейчас (например, обилие функций без аннотаций, длинные docstring'и), — в исключения `[tool.ruff.lint.per-file-ignores]`. План фиксирует точный список после первого прогона `ruff check`.

### 4.4 `.env` / `.env.example` / `.gitignore`

- `.gitignore`: добавить `.env`.
- `.env.example`:
  ```text
  BOT_TOKEN=
  DB_PATH=yaride.db
  # см. app/config.py — все опциональные UX-константы.
  ```
- Команды очистки истории документируются в плане; выполняются владельцем вручную:
  ```bash
  git filter-repo --invert-paths --path .env
  ```
  План включает чек-листы «до/после» (`git log --all --source --remotes -- .env`).
- После очистки истории — `git push --force` на remote по решению владельца. План помечает это явно как опасный шаг.

## 6. Состояние БД

## 7. Тестирование

- Юнит-тест на загрузку настроек: списки парсятся корректно, некорректное значение валит `load_settings()` с понятной ошибкой.
- Юнит-тест на `schema_version`: первая инициализация ставит версию; повторный вызов `init_schema` не меняет.
- Юнит-тест на «старая БД без `schema_version`» (таблицы есть, версии нет): после `init_schema` версия выставлена, миграции не выполнялись повторно (проверяется через факт неизменности таблицы `route_points`).
- `ruff check` и `ruff format --check` в CI-шаге (или, при отсутствии CI, в `README.md` как обязательный шаг до коммита).

## 8. Критерии приёмки

1. Token rotated в BotFather, новый токен лежит в локальном `.env`, который не отслеживается git.
2. `git log --all -- .env` пуст после очистки истории (выполняется владельцем).
3. `.env.example` присутствует, `.gitignore` содержит `.env`.
4. `AGENTS.md` обновлён и отражает новую политику.
5. `ruff check` и `ruff format --check` зелёные.
6. `app/config.py` экспортирует UX-константы; `bot.py` импортирует их вместо литералов.
7. `schema_version` создаётся при старте; повторный старт не вызывает миграций.
8. Все существующие тесты + новые тесты этого этапа — зелёные.
9. Регрессий по существующим сценариям нет.
