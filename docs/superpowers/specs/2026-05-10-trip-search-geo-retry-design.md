# Trip search: повторная геолокация без полной очистки чата

**Статус:** согласовано в brainstorming (2026-05-10).  
**Продукт:** Yaride — Telegram-бот карпула (aiogram 3, SQLite).  
**Область:** шаг выбора точки отправления в сценарии **поиска поездки** при использовании **геолокации** (аналогичный поток создания поездки затрагивается тем же механизмом — см. раздел «Область применения»).

---

## 1. Проблема

Сейчас обработчик геолокации для шага стартовой локальности (`_handle_start_locality_geo` в `app/bot.py`) в начале вызывает **`cleanup_chat`**. Реализация в `ChatUiService.cleanup_chat` (`app/chat_ui.py`):

- удаляет все отслеживаемые сообщения бота из истории чата;
- удаляет **входящее сообщение пользователя** (в т.ч. сообщение с геометкой).

Если подсказки ближайших остановок по координатам нерелевантны (ошибка GPS, «точка за 5 км» и т.д.), пользователь теряет контекст и не может **спокойно повторить** отправку геолокации, не проходя сценарий заново. Сообщение с подсказками и связанный контекст исчезают раньше, чем пользователь подтвердил выбор остановки.

## 2. Цели и ограничения

### Цели

1. При повторной отправке геолокации в том же шаге **обновлять только блок подсказок** (текст + инлайн-кнопки остановок), не выполняя **полную** очистку чата.
2. Сохранять для пользователя возможность несколько раз прислать геолокацию, пока список остановок не станет приемлемым с его точки зрения.
3. Зафиксировать успех гео-ветки прежним правилом: пользователь нажимает **одну из кнопок предложенной остановки** (callback `gxs:...`), после чего выполняется существующий переход в оркестраторе потока.

### Не в рамках этой спецификации

- Изменение алгоритма ранжирования остановок (`nearest_stops_global`, лимиты км).
- Новая отдельная кнопка «Подходит» для подтверждения списка (пользователь явно выбрал вариант **C**: успех = выбор остановки из списка).
- Отдельное приложение или смена канала (Telegram только).

## 3. Предлагаемое поведение

### 3.1 Первая геолокация на шаге

- **Не** вызывать `cleanup_chat` для входящего сообщения с геолокацией при входе в `_handle_start_locality_geo`.
- Отправить одно сообщение бота с текстом подсказки (км по прямой до точки каталога) и инлайн-клавиатурой топа остановок.
- Сохранить в FSM идентификатор этого сообщения (`message_id` в чате), чтобы следующая геолокация могла обновить **именно его**.

### 3.2 Повторная геолокация на том же шаге

- Пересчитать топ остановок по новым координатам.
- Обновить **то же** сообщение бота: `edit_message_text` и при необходимости `edit_message_reply_markup` (или эквивалент через aiogram).
- Если редактирование невозможно (сообщение удалено, контент не меняется по правилам Telegram, ошибка API): удалить **только** предыдущее сообщение блока подсказок (если оно ещё существует) и отправить новое; обновить сохранённый `message_id` в FSM.

### 3.3 Успешный выбор остановки

- Без изменения контракта: пользователь нажимает кнопку остановки → существующий `geo_pick_suggested_start_stop` / `TripFlowOrchestrator.transition_geo_pick_start_stop`.
- После успешного перехода к следующему шагу снять из FSM сохранённый `geo_suggest_message_id` (или эквивалентное поле).

### 3.4 Ветки без списка остановок

Если по координатам не найдено остановок в пределах порога, но определилась локальность (`nearest_locality_from_geo`), или показана ошибка «слишком далеко»:

- По-прежнему **не** использовать полный `cleanup_chat` в начале обработчика гео таким образом, чтобы стирать весь контекст шага; допускается отправка отдельных сообщений с `ReplyKeyboardRemove` и переходом к выбору района, как сейчас по смыслу — но без массового удаления истории из-за первой строки обработчика.

(Точное объединение с текущими ветками `sent_err` / `sent_loc` фиксируется при реализации: либо те же сообщения, либо редактирование единого блока — без противоречия цели «не сносить весь чат».)

## 4. Состояние FSM

Добавить (имена примерные, финально — как в коде):

| Поле | Назначение |
|------|------------|
| `geo_suggest_message_id` | `message_id` сообщения бота с блоком подсказок остановок для текущего шага гео |

При выходе из шага (назад, смена состояния, успешный выбор остановки) — очистка поля.

## 5. Изменения по компонентам (высокий уровень)

### 5.1 `app/bot.py`

- Убрать начальный `cleanup_chat` из `_handle_start_locality_geo` в пользу логики «первая отправка / редактирование».
- При первой геолокации: `answer` + `track_bot_message`; записать id в FSM.
- При повторной: попытка `edit` целевого сообщения; fallback — удалить только старый блок и отправить новый.
- Не удалять входящее сообщение пользователя с геолокацией через полный cleanup в этом обработчике.

### 5.2 `app/chat_ui.py`

- Опционально: вспомогательный метод «удалить только сообщение с известным `message_id`» без очистки всей истории — если не хочется размазывать вызовы `delete_message` по боту.
- Полный `cleanup_chat` не расширять обязательными параметрами, если достаточно **не вызывать** его в гео-обработчике; при необходимости узкой политики удаления — отдельный маленький метод с явным именем (например `delete_message_safe`), без изменения поведения остальных хендлеров.

### 5.3 Поток создания поездки

Обработчик `_handle_start_locality_geo` общий для `mode="search"` и `mode="create"`. Изменения применяются к **обоим** режимам, чтобы поведение «повтор гео без полной зачистки» было одинаковым.

## 6. Ошибки и краевые случаи

- **Несколько подряд геосообщений:** каждое обновляет один и тот же блок (или заменяет его при fallback).
- **Пользователь удалил сообщение с подсказками вручную:** следующая геолокация создаёт новое сообщение; FSM обновляется.
- **Выход «Назад» или смена шага:** сброс `geo_suggest_message_id`, чтобы не редактировать чужое сообщение.
- **Лимит истории `bot_chat_messages`:** новые сообщения по-прежнему через `track_bot_message`; не допускать неконтролируемого роста дубликатов блоков подсказок — при успешном edit старый id тот же.

## 7. Тестирование

- Юнит-тесты без Telegram: при наличии изолированной логики «решить: edit vs new message» — моки FSM и состояние id.
- Ручная проверка в Telegram: две-три геолокации подряд на шаге поиска; убедиться, что сообщение с городом/меню выше не исчезает без необходимости и что выбор остановки ведёт к следующему шагу как сейчас.

## 8. Критерии приёмки

1. На шаге отправления поиска поездки пользователь может отправить геолокацию **несколько раз**; блок с топом остановок **обновляется**, а не исчезает вместе со всей историей из-за `cleanup_chat` в начале обработчика.
2. Выбор остановки из подсказки по-прежнему переводит поток дальше (регрессии нет).
3. Создание поездки по геолокации ведёт себя согласованно с поиском (общий обработчик).
