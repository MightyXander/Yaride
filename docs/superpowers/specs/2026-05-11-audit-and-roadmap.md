# Yaride MVP — аудит и roadmap улучшений

**Дата:** 2026-05-11.
**Контекст:** сессия аудита бота карпула Yaride (aiogram 3.x, Python 3, SQLite, ~17 модулей в `app/`, 32 unit-теста).
**Статус:** roadmap согласован с владельцем проекта. Детальные спеки этапов лежат рядом в той же папке. Планы реализации (`docs/superpowers/plans/...`) пишутся отдельно перед каждым этапом.

---

## 1. Зачем этот документ

Этот документ — общий контракт о направлении развития. Он:

1. Фиксирует слабые стороны проекта на момент 2026-05-11 (раздел 2).
2. Декомпозирует улучшения на пять независимых этапов с явным порядком и зависимостями (раздел 3).
3. Указывает, какие конкретные находки решает каждый этап (раздел 4).
4. Задаёт сквозные правила, общие для всех этапов (раздел 5).
5. Перечисляет осознанно отложенные темы (раздел 6).

Документ не описывает, **как** делать изменения — это работа отдельных спеков и планов.

## 2. Слабые стороны (audit findings)

Каждая находка имеет идентификатор (`H` — high, `M` — medium, `L` — low), краткое описание и точку в коде. Полный текст обоснования и доказательства лежат в логе сессии, здесь — компактная справка для команды.

### Высокий приоритет

- **H1. Живой `BOT_TOKEN` в репозитории.** `.env` отслеживается git, содержит реальный токен; `.gitignore` его не исключает. AGENTS.md ранее просил не ротировать — это решение пересмотрено владельцем 2026-05-11.
- **H2. Гонка при бронировании последнего места.** `BookingRepository.create_booking` (`app/repo.py:497-560`) делает `SELECT seats_booked` → проверку в Python → `UPDATE seats_booked = seats_booked + 1`. Без `BEGIN IMMEDIATE` и без условного `UPDATE ... WHERE seats_booked < seats_total` два параллельных бронирования могут оба пройти.
- **H3. Миграции запускаются на каждой транзакции.** `Database.transaction()` (`app/db.py:50-61`) вызывает `_ensure_extended_schema(conn)` перед каждым SQL. Каждая операция чтения открывает новое sqlite-соединение, проверяет схему, ставит PRAGMA. Нет `schema_version`, нет WAL.
- **H4. Глобальное состояние модуля `bot.py`.** Синглтоны `KEYBOARDS`, `CHAT_UI`, `FLOW_ORCHESTRATOR`, `NAVIGATION_FLOW` на уровне модуля; `global _REPO_FOR_KB` в `run()`. Делает handler'ы нетестируемыми и блокирует второй инстанс в том же процессе.

### Средний приоритет

- **M5. `app/bot.py` ~1700 строк.** 50+ handler'ов, 5 FSM-групп, кастомный календарь, фабрики, инициализация — в одном файле.
- **M6. Дублирование Search/Create в `NavigationFlow`.** `app/navigation_flow.py:75-274` — ветки `search_*` и `create_*` отличаются только префиксом callback, но кода в обе стороны написано почти одинаково. `FLOW_MODE_CFG` уже описывает разницу, но не используется.
- **M7. Фасад `Repo` с 35+ прокси-методами.** `app/repo.py:1052-1240` дублирует API под-репозиториев; handler'ы зовут то через фасад, то напрямую — нет единого правила.
- **M8. Нет индексов под типовые запросы.** `trips(status, trip_date, start_point_id, end_point_id)`, `bookings(passenger_id, status)`, `bookings(trip_id, status)`, `trip_ratings(rated_user_id)` — отсутствуют.
- **M9. Миграции без версии.** 9 разрозненных `_migrate_*` методов + `_fill_route_point_coordinates` дёргается на каждой транзакции. Идемпотентность держится на дисциплине, не на структуре.
- **M10. Полный `cleanup_chat` на каждом шаге.** До 14 HTTP-вызовов Telegram на одну кнопку. Spec `2026-05-10-trip-search-geo-retry-design.md` боролся с симптомом; причина — модель «зачищаем всё, рисуем заново».

### Низкий приоритет

- **L11. `rating_worker` O(N·M).** `RatingRepository.list_pending_rating_prompts` (`app/repo.py:947-1049`) — для каждой пары trip×booking делает 4 `SELECT 1`. Один `LEFT JOIN` дал бы O(1) запрос.
- **L12. Слабое покрытие критичной бизнес-логики.** 0 тестов на `BookingRepository`, `TripRepository`, `RatingRepository`, миграции, handler'ы.
- **L13. Логи без структуры.** `basicConfig(level=INFO)` без ротации, JSON-формата, correlation id.
- **L14. Нет линтера/форматтера.** Стили расходятся между файлами.
- **L15. Захардкоженные UX-константы.** `seats ∈ {2,3,4}` (`app/bot.py:1162`), `price ∈ {100,150,200}` (`app/bot.py:1183`), `GEO_SUGGEST_MAX_KM=85.0`, рабочие часы `range(6,23)` — всё в коде.
- **L16. Graceful shutdown отсутствует.** `asyncio.create_task(rating_prompt_loop())` (`app/bot.py:1707`) без хранения ссылок и отмены.
- **L17. `drop_pending_updates=True`** при старте — обновления за рестарт теряются молча.
- **L18. Идентификация водителя только по формату ВУ.** Осознанное ограничение MVP; фиксируем как «известный долг».

## 3. Roadmap: пять этапов

Этапы упорядочены так, чтобы каждый следующий опирался на уже стабильный фундамент. Каждый — со своим спеком и планом реализации; в реализацию идём по одному этапу за сессию.

| Этап | Тема | Спек | Ориентировочный объём |
|------|------|------|------------------------|
| 0 | Гигиена и основания | `2026-05-11-etap-0-hygiene-and-foundations.md` | 1-2 дня |
| 1 | Данные и устойчивость | `2026-05-11-etap-1-data-resilience.md` | 3-5 дней |
| 2 | Производительность БД | `2026-05-11-etap-2-db-performance.md` | 2-3 дня |
| 3 | Архитектура: распил `bot.py` | `2026-05-11-etap-3-architecture-refactor.md` | 5-7 дней |
| 4 | UX чата | `2026-05-11-etap-4-chat-ux.md` | 2-3 дня |

Сквозная тема **тестов** — не отдельный этап, а дисциплина в каждом: каждое поведение, которое меняется или фиксируется, покрывается unit-тестом до или вместе с реализацией.

### Зависимости между этапами

- Этап 1 опирается на `schema_version` из Этапа 0 (атомарные миграции).
- Этап 2 опирается на единый стартовый прогон миграций из Этапа 1 (нет смысла строить индексы, если миграции дёргаются на каждом SQL).
- Этап 3 не зависит технически от 1 и 2, но рефакторить файл `bot.py` поверх нестабильной схемы и гонок — рискованно; поэтому он после 2.
- Этап 4 (UX чата) последний: `ChatUiService` уже изолирован, и его проще менять, когда handler'ы разбиты по доменам (Этап 3).

## 5. Сквозные правила для всех этапов

- **TDD.** Любое изменение поведения сопровождается unit-тестом. Поведение не подтверждается без теста.
- **Маленькие коммиты.** Каждый шаг — один логический коммит, проходящий тесты.
- **Никаких параллельных рефакторингов.** В рамках этапа делается только то, что описано в его спеке. Сторонние улучшения выносятся в backlog в roadmap.
- **Тесты не удаляются вместе с заменяемым кодом.** Тест меняется или заменяется, но всегда остаётся регрессия.
- **Совместимость БД.** Любая миграция должна корректно отрабатывать на уже существующих `yaride.db`. Тест миграции — обязателен.
- **Без новых внешних зависимостей без согласования.** `ruff` (Этап 0) — единственная согласованная новая зависимость.

## 6. Что осознанно НЕ делаем в этом roadmap

- **Реальная верификация ВУ** (L18). Требует интеграции с ГИБДД/Госуслуги; не для MVP.
- **Anti-abuse / captcha** на регистрацию. На текущем масштабе не нужно.
- **Платежи.** Явно отказано в README.
- **i18n.** Город один, язык один.
- **Замена SQLite на Postgres.** AGENTS.md прямо просит «no Docker, no external databases».
- **Замена aiogram polling на webhook + production deploy.** Отдельный проект.
- **`drop_pending_updates`** (L17). Пока бот рестартуется редко, накопленные апдейты не критичны.
- **Восстановление истории git от `BOT_TOKEN`** через `git filter-repo` — выполняется как часть Этапа 0, но **не** будет автоматизировано в плане: владелец делает это вручную, чтобы не потерять историю случайно.

## 7. Запись о смене политики AGENTS.md

AGENTS.md на 2026-05-11 говорит: «The `.env` file is committed to the repo with a bot token. Treat it as a development convenience; do not rotate or remove it.»

Решением владельца 2026-05-11 эта политика отменена в пользу ротации токена и удаления `.env` из git-истории. **Этап 0 включает обновление AGENTS.md, чтобы зафиксировать новую политику явно.** Иначе будущие сессии будут получать противоречивые инструкции.

## 8. После этапа 4: backlog (не в текущем roadmap)

Темы ниже осознанно не входят в этапы 0–4, но логично идут **после** закрытия UX-чата (этап 4), когда стабильны данные, архитектура и нагрузка на Telegram API.

| Тема | Зачем | Зависимости / заметки |
|------|--------|------------------------|
| **L17 — политика `drop_pending_updates`** | Пересмотреть: при редких рестартах потеря очереди приемлема; при частых — включить накопление или graceful drain | После этапа 1 (корректный shutdown) проще сравнить сценарии |
| **Полный structured logging** | JSON на stdout, ротация файлов, уровни по модулям; не только `extra` на критичных путях | После этапа 1; не смешивать с рефакторингом handler'ов |
| **CI (GitHub Actions / аналог)** | `ruff check`, `ruff format --check`, `unittest` на каждый PR | После этапа 0 (ruff уже есть) |
| **Нагрузочный smoke** | Скрипт или заметка: N параллельных пользователей / синтетические trip, без внешних сервисов | После этапов 1–2 |
| **Документация оператора** | Короткий runbook: ротация токена, бэкап `yaride.db`, где смотреть логи | В любой момент; усиливается после этапа 1 |
| **Расширение MVP (город/язык)** | i18n, второй регион — только если появится продуктовая необходимость | Не планируется в таблице отложенного (раздел 6) до смены стратегии |

Ничего из этого не блокирует завершение этапов 0–4; порядок внутри backlog гибкий, кроме явных зависимостей в колонке.

## 9. Критерии «roadmap закрыт» (этапы 0–4)

Использовать как финальную приёмку всей волны, не дублируя детальные критерии из спеков этапов.

1. **Секреты:** `.env` не в git; актуальная политика в `AGENTS.md` и ссылка на `.env.example`.
2. **Данные:** нет гонки на последнем месте; миграции линейные и один раз на старт; WAL и осмысленное использование соединения (как в спеке этапа 1).
3. **Наблюдаемость минимум:** логи критичных действий с идентификаторами; shutdown без «висящих» задач.
4. **БД:** индексы и один SQL для `list_pending_rating_prompts` (этап 2); при росте данных запросы не деградируют по плану из спека.
5. **Код:** `bot.py` — тонкий runner; нет глобальных синглтонов для зависимостей; `NavigationFlow` без дублирования веток; `Repo` — тонкий контейнер под-репозиториев.
6. **UX Telegram:** переходы во flow не порождают каскад удалений как основную модель (этап 4).
7. **Тесты:** регрессия зелёная; новое или изменённое поведение покрыто unit-тестами по правилам раздела 5.
