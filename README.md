# Yaride MVP (Telegram carpool bot)

MVP карпулинг-бота для Ярославля и ближайших населённых пунктов.

## Что реализовано

- Регистрация пользователя и выбор роли (`водитель` / `пассажир`).
- Поиск/просмотр поездок доступен **обеим ролям**.
- Создание поездок доступно только водителю.
- Бронирование доступно только пассажиру.
- Пассажир может отменить свою бронь с причиной:
  - место автоматически освобождается;
  - водителю уходит уведомление с причиной отмены.
- Полноценный выбор даты через `aiogram-calendar` (месяцы/годы/дни).
- Поиск поездок: маршрут + дата (без обязательного выбора времени).
- Смена роли доступна, если у водителя нет активных поездок на выбранную дату.
- Справочник точек маршрута:
  - популярные остановки Ярославля (по 1-2 для каждого района);
  - близлежащие города/сёла без остановок.

## Быстрый старт

1. Установить зависимости:
   - `pip install -r requirements.txt`
2. Создать `.env` на основе `.env.example` (см. файл в репозитории):
   - `cp .env.example .env`
   - Открыть `.env`, вписать `BOT_TOKEN=<токен от BotFather>`.
   - При желании — переопределить UX-константы (см. `app/config.py`).
3. Запустить:
   - `python main.py`

## Веб-админка (модерация)

Локальный веб-интерфейс модерации поверх того же `yaride.db` (модуль `admin/`). Все правки идут
через `app/repo.py` и `app/services/admin_service.py`, поэтому инварианты (места, рейтинг, FK) и
каскады сохраняются. Действия пишутся в журнал `admin_audit_log`; отмена поездки и бан уведомляют
пользователей через бота.

1. Установить зависимости: `pip install -r requirements.txt` (для тестов — `requirements-dev.txt`).
2. (Опц.) задать в `.env` постоянный секрет сессий `ADMIN_SESSION_SECRET=<случайная строка>`.
3. Создать учётную запись администратора (пароль вводится интерактивно):
   - `py -3 -m admin.cli create-admin <логин>` (Linux/macOS: `python3 -m admin.cli ...`)
4. Запустить интерфейс:
   - `py -3 -m admin` — поднимет `http://127.0.0.1:8000` (адрес/порт меняются через `ADMIN_HOST`/`ADMIN_PORT`).

Возможности: сводка, поездки (правка цены/мест/даты/статуса, принудительная отмена), брони (просмотр),
пользователи (правка профиля, бан/разбан), оценки (правка/скрытие отзыва, удаление с пересчётом рейтинга),
точки маршрута (CRUD с защитой удаления при ссылках из поездок), журнал аудита.

Забаненный пользователь отсекается `BanMiddleware` (`app/middlewares.py`) до доменных хендлеров бота.

## Основные кнопки

- `Найти поездки` — подбор поездок по отправлению, прибытию и дате.
- `Создать поездку` — только для водителя.
- `Мои брони` — список броней пассажира и отмена активных броней.
- `История поездок` — созданные поездки водителя (список).
- `Управление` — только для водителя: брони, отмена поездки, порог рейтинга пассажиров.

## Архитектура и читаемость

Проект организован по слоям, чтобы каждый модуль отвечал за одну зону ответственности:

- `app/bot.py` — обработчики Telegram-сценариев и UI.
- `app/repo.py` — слой доступа к данным и бизнес-правила.
- `app/db.py` — подключение, транзакции и инициализация схемы SQLite.
- `app/ui.py` — фабрика клавиатур (`KeyboardFactory`) для всех UI-кнопок.
- `app/chat_ui.py` — сервис чата (`ChatUiService`) для очистки, перерисовки и истории сообщений.
- `app/trip_flow.py` — оркестратор сценариев (`TripFlowOrchestrator`) для общих переходов поиска/создания.
- `app/navigation_flow.py` — централизованная логика навигации “Назад” (`NavigationFlow`) для callback и reply-кнопок.
- `app/formatting.py` — форматирование дат/времени и строк поездок.
- `app/seeds.py` — справочник маршрутов для первичного наполнения БД.
- `app/config.py` — загрузка и валидация настроек из окружения.

В `app/repo.py` применён ООП-подход:

- Фасад `Repo` сохраняет единый API для обработчиков.
- Специализированные классы (`UserRepository`, `RouteRepository`, `TripRepository`, `BookingRepository`) разделяют ответственность по доменам.
- Общие служебные операции вынесены в базовый класс `_BaseRepository`.
- UI-составляющие также разделены по объектам: `KeyboardFactory` отвечает за клавиатуры, `ChatUiService` — за жизненный цикл сообщений.
- В `app/bot.py` сценарные переходы поиска/создания централизованы в `TripFlowOrchestrator`, что уменьшает дублирование хендлеров.
- Для ключевой логики оркестратора добавлены unit-тесты в `tests/test_trip_flow.py` (без запуска Telegram).
- Для навигации назад добавлены unit-тесты в `tests/test_navigation_flow.py`.

Такой подход уменьшает связность, упрощает чтение кода и безопасное внесение изменений.

## Ограничения MVP

- БД SQLite.
- Без геолокации и GPS, только по заранее заданным точкам.
- Без платежей (цена — только как компенсация расходов).

## Индексы и запросы

После этапа 2 в SQLite добавлены индексы под типовые выборки: открытые поездки по статусу/дате/маршруту (`trips`), активные брони по пассажиру и по поездке (`bookings`), оценки и напоминания об оценках (`trip_ratings`, `rating_prompts_sent`). Список имён — в `app/db.py` (`_ensure_performance_indexes`). Напоминания об оценках собираются одним SQL в `RatingRepository.list_pending_rating_prompts`. Синтетический замер до/после отключения этих индексов: `py -3 scripts/benchmark_db.py` (результаты накапливаются в `scripts/benchmark_baseline.txt`).

## Разработка

### Lint и format

Перед коммитом:

```bash
py -3 -m ruff check .
py -3 -m ruff format --check .
```

Авто-исправления:

```bash
py -3 -m ruff check . --fix
py -3 -m ruff format .
```

### Запуск тестов

```bash
py -3 -m unittest discover -s tests -v
```

### Версия схемы БД

Текущая версия — в `app.db.SCHEMA_VERSION`. Таблица `schema_version` в `yaride.db` хранит фактически применённую версию. Линейные миграции по версиям подключаются в Этапе 1 (см. `docs/superpowers/specs/2026-05-11-etap-1-data-resilience.md`).
